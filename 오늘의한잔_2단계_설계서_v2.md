# 오늘의 한잔 — 2단계 설계서 v2 (비회원 로컬 저장 기준)

> 1단계 프로토타입에서 검증된 결재형 UI·추천 로직을, **회원가입 없이 기기에 저장되는(local-first)** 실제 웹 서비스로 전환하기 위한 설계 문서. 개인정보보호법 리스크를 최소화하면서 핵심 가치(오늘 조건에 맞는 술집 추천)를 가장 빠르게 검증하는 것이 목표다.

---

## 1. 방향 결정 요약

v1은 회원가입·서버 DB 없이 간다. 개인 기록(가본 곳·위시·취향 퀴즈·평점·태그)은 전부 사용자 브라우저에 저장하고, 공용 업장 풀만 서버에서 읽기 전용으로 제공한다. 이메일·비밀번호·계정을 받지 않으므로 개인정보 수집·이용 동의, 제3자 제공 동의, 마케팅 수신 동의가 v1에서는 불필요하다. 단 "개인정보 이슈 0"은 아니다 — 호스팅사가 접속 로그(IP)를 남길 수 있고, 광고·분석툴을 붙이는 순간 행태정보 이슈가 다시 생긴다. 그래서 v1에는 광고·분석툴을 붙이지 않고, "데이터를 서버에 저장하지 않고 기기에만 저장한다"는 간단한 안내 페이지를 둔다.

추천사유는 시스템(제3자)이 객관 사유를 대는 **공문서체 타인추천형**으로 통일한다. 콜드스타트는 가입 퀴즈 5문항 + 공용 풀(322곳의 객관 정보만) 결합으로 푼다.

## 2. 저장 구조 (local-first)

핵심 원칙은 "개인 기록은 기기 밖으로 나가지 않는다, 공용 풀은 객관 정보만 읽기 전용으로 빌려준다"이다.

| 영역 | 저장 위치 | 개인정보 리스크 |
| --- | --- | --- |
| 공용 업장 풀 (322곳, 객관 정보만) | 서버 정적 파일(JSON) · 읽기 전용 | 낮음 |
| 취향 퀴즈 5문항 응답 | 브라우저(IndexedDB) | 낮음 |
| 가본 곳 / 위시 / 평점 / 태그 | 브라우저(IndexedDB) | 낮음 |
| 추천 계산 | 브라우저 내부에서 처리 | 낮음 |
| 계정 / 이메일 | 없음 | 없음 |
| 광고 / 분석툴 | 붙이지 않음 | 낮음 |

공용 풀에는 업장명·카테고리·구·동·주소 같은 객관 정보만 적재한다. 1단계에서 팀장님이 매긴 만족도·재방문·좋았던점·아쉬운점 같은 주관 평가는 공용 풀에 넣지 않는다. 그래야 한 사람의 취향이 신규 사용자에게 정답처럼 강요되지 않고, 타인추천형 사유 톤과도 일관된다.

저장 기술은 IndexedDB를 권장한다. 데이터량이 늘어도 안정적이고, 구조화된 기록(업장 목록·태그·퀴즈)을 다루기에 localStorage보다 적합하다.

## 3. 데이터 유실 보완 — 내보내기 / 가져오기

local-first의 약점은 명확하다. 기기를 바꾸거나 브라우저 데이터를 지우면 기록이 사라지고, 여러 기기 동기화가 안 된다. 이를 강제 가입 없이 완화하기 위해 **내보내기/가져오기** 기능을 v1에 넣는다.

내보내기는 사용자의 기기 내 데이터(퀴즈 응답 + 등재한 업장 전체)를 JSON 파일 하나로 떨어뜨린다. 가져오기는 그 파일을 다시 불러와 복원한다. 기기 이전·백업을 사용자가 수동으로 할 수 있게 하는 가벼운 안전장치다. 함께 "브라우저 데이터 삭제/초기화" 버튼도 둔다(사용자의 데이터 통제권 보장).

## 4. 취향 퀴즈 (첫 진입 시 5문항)

회원가입은 없지만, 첫 진입 시 퀴즈 5문항으로 개인 취향 벡터를 만들어 콜드스타트를 메운다. 아카이브 축과 추천 질문이 같은 자(尺)를 쓴다는 1단계 원칙을 유지하므로, 5문항이 그대로 추천 벡터가 된다.

문항 1. 술자리는 주로 누구와 가나요? — 연인 / 직장동료 / 친구 / 혼자 / 가족(복수 선택 가능). 동행자는 유형만 받고 실명 입력은 받지 않는다.

문항 2. 술집에서 가장 중요하게 보는 것은? — 메뉴 구성 다양 / 가격 대비 만족도 / 안주 품질 / 분위기 / 2차 연계 용이.

문항 3. 가장 피하고 싶은 요소는? — 혼잡도 / 소음 / 위생 / 높은 객단가 / 긴 대기시간.

문항 4. 선호하는 술집 성향은? — 노포·로컬 정취 / 트렌디·분위기 / 가성비 실속 / 조용한 1인 가능.

문항 5. 자주 가거나 관심 있는 지역은? — 서울 구 다중 선택. 데이터가 0인 초기에도 지역 필터가 작동하도록 하는 시드.

## 5. 추천 로직 — 공용/개인 가중치 전환

추천 후보는 두 출처를 합친다. 기기 내 본인 기록(위시 + 방문함 중 재방문≠아니오)과 공용 풀(객관 정보만)이다. 점수는 출처별로 다르게 매긴다.

본인 방문함(검증 업장)은 가장 높은 가중치를 받는다. 검증 보너스 + 오늘 조건/퀴즈 매칭 + 만족도 반영. 본인 위시는 등재 의사 보너스 + 오늘 조건 매칭. 공용 풀은 만족도가 없으므로 퀴즈 벡터와 오늘 조건의 객관적 부합도(카테고리·지역·선호유형)만으로 점수를 낸다.

전환의 핵심은 감쇠계수다. 본인 등재 수가 늘수록 공용 풀 점수에 곱하는 계수를 낮춘다(시작값: 계수 = max(0.3, 1 − 본인등재수 / 30)). 등재가 30개를 넘으면 공용은 보조로 물러나고 추천이 개인 데이터 중심으로 이동한다. 즉 퀴즈가 콜드스타트를 메우고, 등재가 그것을 개인화로 갈아끼우는 구조다. 30이라는 기준은 시작값이며 실제 사용하며 튜닝한다.

## 6. 추천사유 — 공문서체 타인추천형

추천 주체는 사용자 본인이 아니라 시스템이다. 그래서 "당신이 전에…"가 아니라 객관 사유를 공문서체로 댄다. 출처별 문구 패턴은 다음과 같다.

본인 방문함(검증) — "본 업장은 귀하의 재방문 의사 등재 건으로, 우대조건 '○○' 부합에 따라 추천 상신함."

본인 위시 — "귀하가 방문 예정으로 등재한 건으로, 금일 조건에 부합하여 우선 검토 대상에 포함함."

공용 풀 — "공용 업장 명부 등재 건으로, 취향 진단 결과 '○○' 선호에 부합하여 추천함."

반려(제외) — "반려사유 '○○'에 해당하여 금일 추천 대상에서 제외함."

## 7. 개인정보 관련 설계 원칙 (v1)

회원가입이 없어도 신뢰를 위해 간단한 개인정보 안내 페이지를 둔다. 핵심 문구는 "본 서비스는 회원가입·개인정보 수집 없이, 데이터를 사용자 기기(브라우저)에만 저장합니다. 데이터 삭제는 [초기화] 버튼으로 직접 하실 수 있습니다." 수준이면 충분하다.

설계상 지켜야 할 것들. 위치는 GPS 자동 인식이 아니라 구 수동 선택을 유지한다(위치정보법 이슈 회피). 동행자는 유형만 받고 실명 입력은 막는다. 한줄평 자유입력 칸에는 "타인의 이름·연락처·직장명 등 개인정보는 입력하지 마십시오" 안내를 두고, MVP에서는 태그형 입력을 우선한다(자유 메모에 타인 정보가 섞일 위험 차단). 이름·전화번호·생년월일·성별·주소·직장명·정확한 위치는 받지 않는다(최소수집 원칙).

## 8. 단계별 로드맵

| 단계 | 내용 | 개인정보 처리 |
| --- | --- | --- |
| v1 | 비회원 로컬 저장 · 광고/분석툴 없음 · 내보내기/가져오기 | 간단 안내 페이지(수집 거의 없음) |
| v1.5 | 선택적 이메일 백업 계정(원하는 사람만 가입, 다기기/백업용) | 가입자에 한해 처리방침 정식 적용 |
| v2 | 로그인 기반 동기화 · 서버 DB(Supabase RLS) · export/delete | 처리방침 정식 게시 · 처리위탁/국외이전 고지 |
| v3 | 광고/제휴/CRM 추가 | 행태정보·쿠키 고지 · 별도 선택 동의 |

v1.5의 핵심은 "기본은 비회원, 데이터 백업·다기기 사용을 원하는 사람만 이메일을 연결"하는 것이다. 가입을 강제하지 않으면서 니즈 있는 사용자만 개인정보 처리 범위로 들어오게 한다. Supabase·서버 DB는 v2에서 등장하므로, v1에서는 인증·DB·국외이전 같은 무거운 작업이 전혀 필요 없다.

## 9. v1 빌드 순서

① 1단계 결재 UI(현재 프로토타입)를 기반으로, 박힌 데이터를 분리한다. ② 공용 풀 322곳을 객관 정보만 담은 정적 JSON으로 만들어 읽기 전용으로 둔다. ③ 첫 진입 시 취향 퀴즈 5문항 온보딩 + 기기 저장. ④ 가본 곳/위시 등재·수정을 IndexedDB에 저장하도록 연결한다. ⑤ 추천 후보를 기기 기록 ∪ 공용 풀에서 조회하고 감쇠계수·공문서체 사유를 적용한다. ⑥ 내보내기/가져오기 + 초기화 버튼 + 개인정보 안내 페이지를 붙인다.

이 작업은 실제 코드 빌드이므로 Claude Code로 진행한다. 본 설계서를 Claude Code 세션에 넘기면 ①~⑥ 순서대로 진행할 수 있다.

## 시작 전 확정 사항

본 v2 설계서는 다음을 확정 전제로 작성됐다. 비회원 로컬 저장으로 시작 / 공용 풀은 객관 정보만 / 감쇠 기준 등재 30개(추후 튜닝) / 광고·분석툴 v1 미적용 / 위치는 구 선택·동행자 유형만·한줄평 주의문구. 변경하고 싶은 항목이 있으면 빌드 착수 전에 조정한다.

(개인정보 관련 사항은 일반 정보이며 법률 자문이 아니다. 실제 공개 전에는 개인정보보호위원회(privacy.go.kr) 자료 또는 전문가 확인을 권장한다.)
